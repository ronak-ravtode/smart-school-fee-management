## 1. Project Context

SmartSchoolFee is a **School FinTech fee management system**.

- **Dynamic Fee Engine**: FeeTypes with configurable penalty/waiver rules (JSON). FeeStructures map fee types to class+section.
- **Omnichannel Payments**: UPI, Cash, Cheque. Transaction tracking with receipt generation.
- **Student Fee Ledger**: Real-time financial records — total owed, waivers, partial payments, overdue detection.

## 4. Coding Standards

### TypeScript
- **Zero `any` types.** Use `unknown` and narrow with type guards or Zod parsing.
- All functions must have explicit return types.
- Use `interface` for object shapes, `type` for unions/intersections/generics.
- Prefer `readonly` for function parameters and immutable data.
- Use `as const` for literal tuples and enums.

### Zod Validation
- **Every API endpoint** must validate its request body/params/query with a Zod schema.
- Validate environment variables at startup with a Zod schema.
- Derive TypeScript types from Zod schemas using `z.infer<typeof schema>`.
- Place validation schemas in the controller file or a dedicated `*.schema.ts` file.

```typescript
// Example: src/controllers/student.controller.ts
import { z } from "zod";

export const CreateStudentSchema = z.object({
  name: z.string().min(1).max(100),
  email: z.string().email(),
  class: z.string().min(1),
  section: z.string().min(1),
  rollNumber: z.string().min(1),
});

export type CreateStudentInput = z.infer<typeof CreateStudentSchema>;
```

### API Response Format
All responses MUST follow this exact structure:

```typescript
// Success (single)
{ "success": true, "data": T, "message": "Resource created" }

// Success (paginated)
{ "success": true, "data": T[], "meta": { "page": 1, "limit": 20, "total": 150 }, "message": "Fetched" }

// Error
{ "success": false, "error": { "code": "STUDENT_NOT_FOUND", "message": "No student found" }, "message": "Failed to fetch student" }
```

### Error Handling
- Define `AppError` class extending `Error` with `statusCode` and `code`.
- Throw `AppError` in services for domain errors.
- Central `errorHandler.middleware.ts` catches all errors and returns standardized JSON.
- Never use `try/catch` without rethrowing or handling — no silent failures.
- Use async/await everywhere. No raw callbacks. No `.then()` chains.

## 5. Database Rules

- **ORM**: Prisma exclusively. No raw SQL unless absolutely necessary for complex aggregations (and only in services).
- **Migrations**: Always `prisma migrate dev` — never manually edit SQL files.
- **Models**: PascalCase (`StudentFeeLedger`). **Tables**: snake_case (`student_fee_ledgers`) via `@map`/`@@map`.
- **Relations**: Always define explicit foreign keys with `onDelete` cascade where appropriate.
- **Indexes**: Add `@@index` on any field used in `where`, `orderBy`, or `include`.
- **Financial Calculations**: Must happen in the **Service layer** using Prisma/TypeScript. Never in database triggers, raw SQL, or computed columns.
- **Decimal Handling**: Use `Decimal` type in Prisma for money fields. Convert to `number` only at the response boundary.

- **NEVER modify `schema.prisma`** without explicit user permission for each change.
